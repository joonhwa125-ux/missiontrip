# Plan: 일정 자동+수동 완료 (Schedule Completion)

> **Summary**: 마지막 일정이 영구 "진행중" 상태에 머무는 문제를 해결하기 위한 일정 완료 기능 기획.
> PRD 목적(안전, 누락 방지)과의 충돌 여부, 운영 엣지 케이스, 조장 UX 영향을 검토한다.
>
> **Author**: Product Manager Agent
> **Created**: 2026-03-25
> **Status**: Draft — CTO 승인 필요

## 1. 문제 정의

### 현재 일정 상태 로직

```typescript
// CLAUDE.md 3.6 일정 상태 판별 로직
function getScheduleStatus(schedule: Schedule): ScheduleStatus {
  if (schedule.is_active) return 'active'         // 현재 진행중
  if (schedule.activated_at) return 'completed'   // 활성화된 적 있음 = 완료
  return 'waiting'                                 // 한 번도 활성화 안 됨 = 대기
}
```

**완료 전환 조건:** `is_active = false` + `activated_at IS NOT NULL`
**완료 전환 시점:** 다음 일정이 활성화되어 `activate_schedule()` RPC가 실행될 때만 가능

### 발생하는 문제

| 상황 | 현재 동작 | 기대 동작 |
|------|-----------|-----------|
| 마지막 일정 (3일차 마지막) | 영구 "진행중" — is_active=true 유지 | 일정 완료 후 "완료" 상태 전환 |
| 모든 조 체크인+보고 완료 시 | 일정 상태 변화 없음 | 자동 완료 전환 (선택적) |
| 운영 종료 후 | 항상 진행중으로 표시됨 | 관리자가 종료 처리 가능 |

## 2. 제안된 구현 방식 PRD 검토

### 2.1 PRD 목적과의 충돌 분석

| 분석 항목 | 결론 | 근거 |
|-----------|------|------|
| 안전(누락 방지)과 충돌 여부 | **충돌 없음** | 일정 완료는 체크인 데이터에 영향 없음. 완료 후에도 기록은 유지됨 |
| "안전 > 편의" 원칙 충돌 | **잠재적 위험 있음** | 자동 완료가 조장 체크인 화면을 강제 닫으면 위험 (상세 3.3항 참조) |
| 1회성 서비스 복잡도 | 낮음 | 단일 DB 컬럼 변경 + 1개 broadcast 이벤트 추가 |
| KWCAG 접근성 | 영향 없음 | 상태 표시 텍스트 변경만 발생 |

### 2.2 "전 조 보고완료" 자동 완료 트리거 검토

**제안:** `submitReport()` 완료 후 `reportedCount === totalGroups` 시 `deactivateSchedule()` 자동 호출

#### 문제점 — scope 필터가 복잡도를 증가시킴

현재 스키마에 `ScheduleScope = "all" | "advance" | "rear"` 가 있다.
"전 조 보고완료" 집계 시 해당 일정의 scope에 맞는 그룹만 카운트해야 한다.

```
scope="advance" 일정 → 선발조만 보고 완료해야 자동 완료
scope="all" 일정     → 전체 조 보고 완료해야 자동 완료
```

scope 필터 없이 totalGroups를 전체 조 수로 계산하면 후발조가 없는 선발조 일정이 영구 미완료가 된다.

#### 핵심 위험: 조장 미보고 시 영구 진행중 문제 (미해결)

| 시나리오 | 결과 |
|----------|------|
| 18개 조 중 1개 조가 실수로 보고 미클릭 | 자동 완료 미발동 — 영구 진행중 |
| 네트워크 문제로 보고 전송 실패 | 오프라인 큐에 있어 sync 미완료 — 자동 완료 미발동 |
| 조장이 앱을 닫은 채 이동 (보고 생략) | 자동 완료 미발동 |

**결론:** "전 조 보고완료" 조건은 **자동 완료의 안정적인 트리거로 부적합**하다.
보고는 조장의 자발적 행동이므로 누락 가능성이 상존한다. 이 조건에 의존하면 마지막 일정이 항상 진행중 상태에 머무는 원래 문제를 해결하지 못할 수 있다.

### 2.3 수동 완료 (종료 버튼) 검토

**제안:** 진행중 카드에 "종료" 버튼 추가 → 확인 모달 → `deactivateSchedule()` 호출

**적합성 평가:**

| 항목 | 평가 |
|------|------|
| 관리자 전용 여부 | 적합 — 일정 관리 권한은 이미 admin 전용 |
| UI 배치 위치 | 검토 필요 — 활성화 버튼과 구분이 명확해야 함 |
| 확인 모달 필요 여부 | 필수 — 실수로 누를 경우 취소 불가 |
| 종료 후 재활성화 가능 여부 | 명세 필요 — 실수 복구 경로 없으면 위험 |

## 3. 운영 시나리오 엣지 케이스

### 3.1 자동 완료 관련 엣지 케이스

| # | 엣지 케이스 | 위험도 | 대응 방안 |
|---|------------|--------|-----------|
| AC-01 | 1개 조 보고 누락으로 자동 완료 미발동 | 높음 | 수동 완료(종료 버튼)를 필수 폴백으로 유지 |
| AC-02 | 자동 완료 직후 체크인 취소 시도 | 중간 | 완료된 일정에는 체크인/취소 불가 처리 필요 |
| AC-03 | scope="advance" 일정에서 후발조 보고 포함 여부 | 높음 | scope 필터 없이 구현 시 자동 완료 오작동 |
| AC-04 | 관리자가 앱 닫힌 상태에서 자동 완료 트리거 | 중간 | 서버사이드 실행이 아닌 클라이언트 트리거라면 발동 불가 |
| AC-05 | 재보고(UPSERT) 시 중복 자동 완료 호출 | 낮음 | deactivateSchedule은 멱등하므로 문제 없음 |

### 3.2 수동 완료 관련 엣지 케이스

| # | 엣지 케이스 | 위험도 | 대응 방안 |
|---|------------|--------|-----------|
| MC-01 | 종료 후 미체크인 인원이 있음을 나중에 발견 | 높음 | 종료 전 미확인 인원 수 확인 모달에 표시 필수 |
| MC-02 | 실수로 종료 클릭 후 복구 필요 | 높음 | 종료 취소(재활성화) 기능 또는 명확한 확인 모달 |
| MC-03 | 종료 직후 조장이 체크인 화면에서 작업 중 | 높음 | 조장 화면 처리 방식 명세 필요 (상세 3.3항) |
| MC-04 | 관리자 2명이 동시에 종료 클릭 | 낮음 | DB 제약으로 멱등 처리 가능 |

### 3.3 조장 UX 관점 — 체크인 화면 자동 완료 시 처리

**이것이 가장 중요한 위험이다.**

시나리오: 조장이 체크인 화면에서 작업 중 → 관리자(또는 자동 완료)가 일정 완료 처리 → `EVENT_SCHEDULE_DEACTIVATED` 수신

현재 `useRealtime.ts`의 `onScheduleActivated` 콜백은 조장 화면에서 토스트 + 이동 링크를 표시한다.
`EVENT_SCHEDULE_DEACTIVATED` 이벤트를 어떻게 처리할지 명세가 없다.

| 처리 방식 | 안전성 | UX | 권장 여부 |
|----------|--------|----|-----------|
| 체크인 화면 강제 종료 | 위험 — 작업 중단 | 나쁨 | **절대 금지** |
| 토스트 알림만 표시 ("일정이 종료되었어요") | 안전 | 양호 | **권장** |
| 읽기 전용 모드 전환 | 안전 | 좋음 | 구현 복잡도 중간 |
| 아무 처리 없음 | 위험 — 상태 불일치 | 나쁨 | 금지 |

**권장:** 토스트 알림 표시 + 체크인 버튼 비활성화(그레이 처리) + 조장이 자발적으로 뒤로가기 선택

## 4. 기능 범위 (MoSCoW)

### Must — 반드시 포함

| # | 요구사항 | 근거 |
|---|----------|------|
| M-01 | 관리자 수동 종료 버튼 (진행중 카드에 배치) | 마지막 일정 영구 진행중 문제의 유일한 안정적 해결책 |
| M-02 | 종료 확인 모달에 미확인 인원 수 표시 | 안전 원칙 — 미확인 인원 있으면 관리자가 인지하고 결정해야 함 |
| M-03 | 종료 후 조장 체크인 화면에 토스트 알림 | 조장이 상태 불일치 상태에서 혼란스럽지 않도록 |

| M-05 | `EVENT_SCHEDULE_DEACTIVATED` Realtime broadcast | 전체 클라이언트 상태 동기화 |
| M-06 | 종료 후 체크인 버튼 비활성화 (조장 화면) | 완료된 일정에 추가 체크인 불가 |

| # | 요구사항 | 근거 |

| S-02 | 종료 취소(재활성화) 기능 | 관리자 실수 복구 경로. 단, 재활성화 = 기존 activateSchedule 재사용 가능 |
| S-03 | 조장 체크인 화면 읽기 전용 모드 전환 | 토스트만으로 불충분하다고 판단 시 |

| C-02 | 관리자 화면에 일정별 소요 시간 표시 | 종료 시각 - 활성화 시각 |

|---|----------|------|

| W-02 | 진행률 100% 시 자동 완료 (보고 없이) | PRD에 보고 절차를 명시했으므로 우회하지 않음 |
| W-03 | 조장이 일정 종료 요청 가능 | 일정 관리는 관리자 전용 — PRD 원칙 |

## 5. 수정된 구현 계획 (제안 보완)

### 5.1 우선 구현: 수동 종료만

자동 완료(전 조 보고완료 트리거)는 **Should**로 내리고,
수동 종료 버튼을 **Must**로 먼저 구현한다.

**근거:**
- 자동 완료는 scope 필터, 클라이언트 사이드 실행, 보고 누락 등 복잡한 엣지 케이스를 동반한다
- 수동 종료는 단순하고 관리자가 명시적으로 제어하므로 "안전 > 편의" 원칙에 부합한다
- 1회성 서비스이므로 과도한 자동화보다 명확한 제어권이 우선이다

### 5.2 새 Server Action: `deactivateSchedule()`

```
위치: src/actions/schedule.ts
권한: admin만 (requireAdmin)
동작:
  1. schedules UPDATE: is_active = false (activated_at은 그대로 유지)
  2. CHANNEL_GLOBAL에 EVENT_SCHEDULE_DEACTIVATED broadcast
     payload: { schedule_id, title }
반환: ActionResult
```

**DB 변경 없음:** `activated_at`이 이미 세팅되어 있으므로 `getScheduleStatus`가 자동으로 'completed'를 반환한다.

### 5.3 Realtime 이벤트 추가

```
상수명: EVENT_SCHEDULE_DEACTIVATED = "schedule_deactivated"
채널: CHANNEL_GLOBAL (전체 수신)
수신 대상: 조장 + 관리자 모두
```

`useRealtime.ts`의 `RealtimeCallbacks`에 `onScheduleDeactivated` 콜백 추가.

### 5.4 조장 화면 처리 (`/group`)

```
onScheduleDeactivated 수신 시:
  1. 체크인 화면에 있는 경우:
     - 토스트: "일정이 종료되었어요. 다음 일정을 확인해주세요"
     - [탔어요!] 버튼 비활성화 (그레이, aria-disabled)
     - 조장이 자발적으로 [← 뒤로] 탭하여 피드로 이동
  2. 일정 피드에 있는 경우:
     - 피드 자동 갱신 (종료된 일정이 dim 처리로 전환)
```

### 5.5 관리자 화면 UI (`/admin`)

```
진행중 카드 → 종료 버튼 위치:
  기존: [전체 현황 보기 >] 버튼만 있음
  변경: [전체 현황 보기 >] | [종료] 버튼 추가

  스타일: 종료 버튼은 빨간 텍스트 (위험 행동 시각적 구분)
         최소 44x44px 터치 영역 (KWCAG 2.5.5)

종료 확인 모달:
  제목: "일정을 종료할까요?"
  내용: "미확인 인원 N명이 있어요. 종료하면 더 이상 체크인할 수 없어요."
        (N = 0이면 "모든 인원이 확인되었어요" 표시)
  버튼: [취소] [종료하기]
```

## 6. 자동 완료 구현 시 추가 요구사항 (Should 구현 시 적용)

자동 완료를 구현하기로 결정할 경우 반드시 아래를 함께 명세해야 한다.

| 항목 | 명세 |
|------|------|
| scope 필터 | 일정의 `scope` 값에 맞는 그룹만 카운트 |
| 실행 위치 | Server Action 내부 (클라이언트 사이드 실행 금지) |
| 중복 방지 | `deactivateSchedule` 내부에서 `is_active` 재확인 후 실행 |
| 실패 시 처리 | 자동 완료 실패해도 보고 자체는 성공으로 처리 |
| 관리자 미발동 알림 | 자동 완료가 발동되지 않아도 관리자가 모름 — 수동 종료 폴백 필수 |

## 7. 성공 기준 (Acceptance Criteria)

| # | 기준 | 검증 방법 |
|---|------|-----------|
| AC-1 | 마지막 일정에서 관리자가 종료 버튼을 누르면 일정이 "완료" 상태로 전환된다 | 수동 테스트 |
| AC-2 | 종료 후 진행중 카드가 완료 카드로 전환된다 (dim 처리, opacity: 0.55) | 수동 테스트 |
| AC-3 | 종료 후 조장 체크인 화면에 토스트 알림이 표시된다 | 2개 브라우저 Realtime 테스트 |
| AC-4 | 종료 후 [탔어요!] 버튼이 비활성화된다 | 수동 테스트 |
| AC-5 | 종료 확인 모달에 미확인 인원 수가 표시된다 | 수동 테스트 |
| AC-6 | 종료는 admin 권한자만 실행할 수 있다 | Server Action 권한 검증 |

## 9. 구현 우선순위 및 타임라인

| 단계 | 내용 | 우선순위 |
|------|------|----------|
| 1단계 | `deactivateSchedule()` Server Action + 상수 추가 | Must |
| 1단계 | `EVENT_SCHEDULE_DEACTIVATED` 상수 + useRealtime 콜백 추가 | Must |
| 1단계 | 관리자 진행중 카드에 종료 버튼 + 확인 모달 | Must |
| 1단계 | 조장 체크인 화면 종료 이벤트 처리 (토스트 + 버튼 비활성화) | Must |
| 2단계 | 자동 완료 (전 조 보고완료 트리거, scope 필터 포함) | Should |
| 2단계 | 종료 취소(재활성화) UI 추가 | Should |
