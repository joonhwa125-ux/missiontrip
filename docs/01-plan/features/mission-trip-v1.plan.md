## 1. User Intent Discovery

### 핵심 문제
이동 시마다 약 200명 중 단 한 명의 인원 누락도 없도록 실시간 확인하는 것.

### 주요 사용자
- **TF장 (관리자)**: 전체 현황 모니터링, 일정 관리. 1~2명.
- **조장**: 조원 체크인 처리, 보고. 18명.
- 두 역할 모두 동등하게 중요.

### 성공 기준
1. 2박 3일 전체 일정에서 인원 누락 0건
2. 종이 체크리스트 대비 체크인 소요 시간 단축
3. 조장·TF장이 "쓰기 편했다, 도움이 됐다"고 느낌

### 제약 조건
- 1회성 서비스 (미션트립 기간만 운영)
- 장애인·비장애인 혼합 → KWCAG 2.2 접근성 필수
- 비용 0원 (무료 tier만 사용)
- 안전 > 편의, 단순함 > 기능 추가

## 2. Alternatives Explored

### 체크인 방식
| 접근 | 설명 | 판정 |
|------|------|------|
| **A: 개별 체크인 (탭탭탭)** | 조장이 조원 한 명씩 확인 | **채택** — 안전성 최고 |
| B: 미탑승자만 체크 | 기본 전원 탑승, 안 탄 사람만 처리 | 기각 — 안전성 저하 |
| C: 전원 탑승 + 예외 처리 | [전원 탑승] 1탭 + 미탑승자 제외 | 기각 — 성급한 완료 위험 |

### UI 구조
| 접근 | 설명 | 판정 |
|------|------|------|
| A: 2탭 구조 | [체크인]/[일정] 탭 분리 | 기각 — 일정 맥락 부족 |
| **B: 피드 드릴다운** | 일정 목록 → 탭 → 체크인 진입 | **채택** — 안전(정확한 일정 인지) + 직관적 |

### 일정 활성화
| 접근 | 설명 | 판정 |
|------|------|------|
| A: 수동 활성화 | TF장이 직접 활성화 | 기각 — 매번 눌러야 함 |
| **B: 자동 + 수동 조정** | 예정 시각에 자동, 필요시 수동 변경 | **채택** — 유연 |
| C: 완전 자동 | 예정 시각에 자동, 수동 개입 없음 | 기각 — 현장 대응 불가 |

## 4. Architecture Overview

### 기술 스택
- Next.js 14 (App Router) + Supabase + Tailwind + shadcn/ui + Vercel

### 화면 구조 (피드 드릴다운)
```
/login     → Google OAuth 로그인
/group     → 조장: 일정 피드 → 체크인 드릴다운
/admin     → 관리자: 현황 탭 + 일정 피드 (관리 기능 포함)
/setup     → 데이터 셋업 (admin 전용)
```

### 조장 화면 흐름
```
일정 피드 (첫 화면)
├── 완료 일정: dim 처리, 완료/전체 표시
├── 진행중 일정: 노란 강조 → 탭 → 체크인 화면 진입
│   ├── 이니셜 동그라미 행
│   ├── 조원 카드 리스트 (탔어요!/불참/취소)
│   ├── 보고 버튼 (하단 고정)
│   └── ← 뒤로 → 일정 피드로 복귀
├── 대기 일정: 일반 표시
└── 전체 조 현황 (하단, 읽기 전용)
```

### 관리자 화면 흐름
```
2탭: [현황] / [일정]
├── [현황] 탭
│   ├── 요약 카드 (보고완료/진행중/시작전)
│   ├── 차량별 조 그리드 (프로그레스 + 배지 + 조장 연락처)
│   └── 드릴다운 → 미탑승 인원 + 불참 처리
├── [일정] 탭 (피드 형태)
│   ├── 완료: dim + opacity
│   ├── 진행중: 강조 + 진행중 pill
│   ├── 대기: [활성화] 버튼
│   ├── 시간 수정 기능
│   └── [+ 일정 추가]
```

### 일정 활성화 흐름
```
예정 시각 도래 → 자동 활성화 → Realtime broadcast
       ↓
TF장 필요시 수동 조정 (시간 변경, 다른 일정 활성화)
```

### 데이터 흐름
```
TF장 일정 활성화(자동/수동)
  → Supabase DB 업데이트
  → Realtime broadcast
  → 조장 앱 일정 피드 갱신

조장 체크인 (탭탭탭)
  → Server Action → Supabase INSERT
  → Realtime broadcast
  → TF장 현황 실시간 반영

조장 보고
  → Server Action → group_reports UPSERT
  → Realtime broadcast
  → TF장 현황 "보고완료" 배지
```

## 5. Key Design Decisions

| 결정 | 선택 | 근거 |
|------|------|------|
| 체크인 방식 | 개별 탭탭탭 | 안전성 최고, 개별 추적 가능 |
| UI 구조 | 피드 드릴다운 | 일정 맥락 인지 → 정확한 체크인 |
| 일정 활성화 | 자동 + 수동 조정 | TF장 부담 최소 + 현장 유연성 |
| 불참 처리 | 조장·관리자 모두 가능 | 조장이 조원 상황 가장 먼저 파악 |
| 불참 전파 | 매번 수동 (자동 전파 없음) | 안전 > 편의, 누락 방지 |
| 공지 | 카카오워크로 대체 | 앱은 인원 체크에 집중 |
| 조원 셀프 체크인 | 제거 | 조장 직접 체크가 원칙 |

## 6. Brainstorming Log

### Phase 1 결정
- 핵심 문제: "인원 누락 방지" 확정
- TF장과 조장 모두 동등하게 중요한 사용자
- 성공 기준: 누락 0건 + 시간 단축 + 사용자 만족 (3가지 모두)

### Phase 2 결정
- 개별 체크인(A안) 채택 — 안전성 최우선
- 피드 드릴다운(B안) 채택 — 기존 2탭 구조에서 변경
- 자동+수동 일정 활성화 채택 — 기존 수동 전용에서 변경

### Phase 3 결정
- 조원 셀프 체크인 제거 (조장 직접 체크 원칙)
- 전원 완료 축하 화면 후순위 (기능에 영향 없음)
- /setup은 1차 포함 (200명 수동 입력 불가)

### Phase 4 결정
- 피드 드릴다운 구조에서 현재 일정 강조 (노란 테두리)
- 일정 자동 활성화 + 수동 조정 방식 확정
