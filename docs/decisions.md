# 설계 결정 기록

> CLAUDE.md(현재 스펙)의 "왜?"를 기록한다. 새 세션에서 결정을 되돌리려면 이 문서를 먼저 확인한다.

---

## 운영 모델

| 결정 | 이유 |
|---|---|
| 조장만 체크인 처리 (셀프 체크인 없음) | 약 180명 조원 중 상당수가 장애인. 앱 사용을 강제할 수 없음. 조장 18명이 대리 체크인 |
| 조 = 버스 1:1 매핑 | 차량 10대, 차량당 1~2 조장. 별도 버스 뷰 불필요 — 조 카드 = 차량 카드 |
| 공지 기능 제거 → 카카오워크 대체 | 참가자 180명은 앱 사용 안 함. 앱 내 공지는 조장 18명에게만 도달. 카카오워크가 전체 도달 가능 |
| member 역할 → /no-access | 조원은 앱 접근 불필요. 실수로 로그인해도 안내 화면만 표시 |

## 데이터 설계

| 결정 | 이유 |
|---|---|
| 장애 여부 컬럼 없음 | 개인정보 최소화. 접근성 갖춘 단일 UI로 모든 사용자 지원 |
| 비상연락 = 이름 + phone + 소속 조 | 이메일은 비상연락에 불필요 |
| group_reports 테이블 (보고 영속화) | Realtime broadcast는 휘발성. 관리자 새로고침 시 보고 상태 유지 필요 |
| is_active UNIQUE INDEX | 동시에 2개 일정 활성화 방지. boolean 단독 사용 금지 |
| is_absent 컬럼 | 불참 인원을 "확인 완료"로 처리하여 전원 완료를 차단하지 않음 |
| checked_by에서 'self' 제거 | 셀프 체크인 폐기. 조장/관리자만 체크인 처리 |

## 시트 파싱 (Setup)

| 결정 | 이유 |
|---|---|
| 이메일 컬럼 제거 (4컬럼) | 조장/관리자는 이름 기반 자동 생성 (`name@도메인`), 조원은 `nologin` placeholder. 사용자가 이메일을 직접 입력할 필요 없음 |
| 조구성 시트 제거 (2시트) | 참가자 시트의 "소속조" 고유값에서 자동 추출. 별도 시트 관리 불필요 |

## UI/UX

| 결정 | 이유 |
|---|---|
| 전체 현황 버튼 → 진행중 카드 내부 하단 | 전체 현황은 "현재 일정"의 조별 상태. 일차 헤더가 아닌 해당 일정 카드에 종속시키는 게 정보 계층상 정확 (Dual-action card 패턴) |
| 관리자 2탭→단일 화면 (일정+현황 통합) | 현황은 항상 특정 일정에 종속된 정보. 별도 탭으로 분리하면 맥락 끊김. 조장 뷰와 동일한 카드→바텀시트 패턴으로 일관성 확보 |
| 요약 영역 탭 → 바텀시트 (KPI 드릴다운) | 카드 전체 탭보다 의도가 명확. "보고 3/10조" 숫자를 보고 "어디가 안 됐지?" 드릴다운하는 자연스러운 흐름 |
| 완료 카드 하단 정렬 | 조장 뷰의 조원 카드와 동일 패턴. 다음 할 일이 항상 위에 → 스크롤 불필요. 모든 일정 완료 시 타임라인 순서 자동 복원 |
| 일차 탭 필터 (관리자+조장 공통) | 3일 × 3~5개 일정 = 최대 15개 카드. 스크롤 문제 해결. 조장도 다른 일차 일정 확인 필요 (읽기 전용) |
| 제목 헤더 제거 | 화면 공간 절약. 접근성은 HTML `<title>`로 충족. 관리자/조장 구분은 콘텐츠 자체가 다르므로 제목 불필요 |
| 대기 카드 [활성화] 버튼 직접 동작 | 대기 카드는 요약 데이터 없음 → 바텀시트 드릴다운 대신 버튼 직접 동작이 자연스러움 |
| 보고 버튼 disabled 금지 | KWCAG 2.5.5 + 미완료 인원 있어도 보고 가능해야 함 (긴급 출발 상황) |
| 확인 모달 없이 즉시 체크인 | 현장 속도 우선. 취소만 모달 확인 |
| 보고 토스트 5초 | WCAG 2.2 상태 메시지 충분 읽기 시간 (접근성 기준) |
| line-through 금지 | 완료 카드에 취소선 사용하면 장애인 사용자에게 혼란 |
| 색상만으로 상태 구분 금지 | KWCAG 1.3.3 감각적 특성. 색상 + 텍스트 + 아이콘 3중 표시 |

## 기술

| 결정 | 이유 |
|---|---|
| Realtime broadcast만 (Polling 금지) | 조장 20명 동시 접속 시 Polling은 DB 부하 폭발. Broadcast는 서버 부하 없음 |
| Server Action만 (클라이언트 직접 DB 금지) | RLS 우회 방지 + 권한 검증 일원화 |
| checkin_updated → group + admin 채널 둘 다 | 관리자도 실시간으로 체크인 현황 확인 필요 |
| 일정 자동 활성화 (클라이언트 타이머) | 서버 cron 불필요. 관리자 앱이 열려있는 동안만 동작 |
